# 🔧 Інформаційна система супроводу заявки на ремонт: від приймання до закриття

---

## 📋 Опис проекту

**Service Center** — це веб-застосунок, розроблений на базі фреймворку Django, призначений для автоматизації та цифровізації процесів сервісного центру з ремонту техніки. Система дозволяє керувати заявками на ремонт, відстежувати стан пристроїв клієнтів, вести базу клієнтів та аналізувати ефективність роботи центру.

### Мета проекту

Основна мета — створення інформаційної системи, яка замінює паперовий документообіг та ручний облік у сервісному центрі на ефективний цифровий інструмент.

---

## ⚙️ Функціональні можливості

### 👥 Управління клієнтами
- Реєстрація нових клієнтів та ведення їхніх профілів
- Перегляд історії звернень кожного клієнта
- Детальна картка клієнта з контактними даними

### 🛠️ Управління ремонтами
- Створення та редагування заявок на ремонт
- Прийом пристроїв (пристрій, несправність, клієнт)
- Відстеження статусу ремонту в реальному часі
- Прив'язка ремонту до конкретного пристрою та клієнта
- Підтвердження видачі / завершення ремонту

### 📊 Аналітика та звітність
- Дашборд з ключовими показниками роботи центру
- Статистика по виконаних ремонтах
- Аналітика завантаженості та ефективності
- Експорт даних (Excel / PDF через `openpyxl` та `reportlab`)

### 🔐 Авторизація та ролі
- Система входу / виходу для персоналу
- Профіль користувача
- Захищені маршрути для авторизованих співробітників

## 🚀 Встановлення та запуск

### Вимоги

- Python 3.10+
- pip

### Крок 1 — Клонувати репозиторій

```bash
git clone https://github.com/Wasd2w/service_center.git
cd service_center
```

### Крок 2 — Створити та активувати віртуальне середовище

```bash
python -m venv venv

# Windows
venv\Scripts\activate

# Linux / macOS
source venv/bin/activate
```

### Крок 3 — Встановити залежності

```bash
pip install -r requirements.txt
```

### Крок 4 — Застосувати міграції

```bash
python manage.py migrate
```

### Крок 5 — Заповнити базу тестовими даними (опціонально)

```bash
python manage.py seed_data
```

### Крок 6 — Запустити сервер розробки

```bash
python manage.py runserver
```

Відкрийте браузер: [http://127.0.0.1:8000](http://127.0.0.1:8000)

## 🗃️ Моделі даних

### `Client` — Клієнт
Зберігає інформацію про клієнтів сервісного центру: ім'я, контактні дані, історія звернень.

### `Device` — Пристрій
Пристрій, що приймається на ремонт: тип, марка, модель, серійний номер.

### `Repair` — Заявка на ремонт
Основна сутність системи: прив'язана до клієнта та пристрою, містить опис несправності, статус, дати прийому та видачі. Реалізує машину станів через `STATUS_TRANSITIONS`.

### `Part` — Запчастина
Запчастина, використана при ремонті: назва, кількість, ціна за одиницю.

### `RepairComment` — Коментар
Внутрішній коментар персоналу до заявки на ремонт.

## 📚 Документування коду

Проєкт використовує **Google Style Docstrings** для Python та **Sphinx** для генерації HTML-документації.

> ⚠️ **Правило для команди:** документація оновлюється **разом із кодом** у тому самому коміті. Код без документації не приймається до гілки `main`.

### Що документувати обов'язково

- Усі публічні функції та методи (`services.py`, `views.py`, `models.py`)
- Усі класи та їхні атрибути
- Складні алгоритми та бізнес-правила (машина станів, обмеження переходів)
- Нетривіальна логіка в `save()`, `clean()`, property-методах

### Що можна не документувати

- Приватні допоміжні функції з очевидною логікою (`_labels`)
- Тривіальні `__str__` якщо повернення очевидне

### Шаблон для функцій

```python
def my_function(param1: str, param2: int) -> bool:
    """Коротко що робить функція (одним реченням).

    Детальний опис якщо потрібно: пояснення алгоритму,
    бізнес-правил або важливих побічних ефектів.

    Args:
        param1: Опис першого параметру.
        param2: Опис другого параметру.

    Returns:
        Опис того що повертається.

    Raises:
        ValidationError: Коли і чому виникає помилка.

    Example:
        >>> result = my_function("test", 42)
        True
    """
```

### Шаблон для класів

```python
class MyModel(models.Model):
    """Коротко що представляє цей клас.

    Детальний опис: призначення, зв'язки з іншими моделями,
    архітектурні рішення (чому саме так).

    Attributes:
        field_name: Опис поля та його бізнес-значення.
    """
```

### Шаблон для Django views

```python
@login_required
def my_view(request: HttpRequest, pk: int) -> HttpResponse:
    """Обробляє GET/POST запити для [опис сторінки].

    GET: повертає відрендерений шаблон.
    POST: валідує форму, зберігає, перенаправляє.

    Args:
        request: Django HttpRequest об'єкт.
        pk: Первинний ключ об'єкта.

    Returns:
        HttpResponse або redirect після POST.
    """
```
